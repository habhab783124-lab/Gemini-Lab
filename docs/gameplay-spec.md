# Gemini-Lab 玩法规范

Updated: 2026-04-21

## 文档状态
这是一份基于当前 README、`Assets/README.md` 与 `Assets/plan.md` 整理出的玩法规范草案。

它描述的是项目想要做成的目标体验，不代表这些功能现在都已经实现。

## 产品一句话
Gemini-Lab 是一个基于 Unity 2D 与 OpenClaw Gateway 的 AI 桌宠项目，目标是做出一个能在公寓场景与桌面 Overlay 间切换、具备长期陪伴感和性格演化能力的虚拟宠物。

## 核心体验支柱
1. 陪伴感：宠物不是静态挂件，而是会主动回应、主动行动、持续成长的陪伴体。
2. 状态驱动：宠物的行为由状态值和性格值共同驱动，而不是单纯由脚本时间轴驱动。
3. 双形态运行：既能在游戏内公寓场景活动，也能以桌面 Overlay 形态陪伴用户。
4. 环境反馈：家具、场景、前台应用、旅行结果都会反过来影响宠物状态与性格。

## 当前目标平台
- Unity 2022.3 LTS
- 2D 正交视角
- 优先桌面环境，目标覆盖 Windows，README 里也保留了 macOS 方向

## 核心系统

### 1. 宠物状态值系统
宠物至少拥有以下状态值：
- 心情
- 饱食
- 精力
- 经验值

这些数值会决定：
- 宠物当前更倾向做什么
- 是否会进入睡眠或工作等状态
- AI 回复时的上下文基调

### 2. 性格演化系统
当前规划的性格维度包括：
- 正直
- 冷静
- 邪恶
- 善良
- 勇敢
- 懦弱
- 害羞

设计目标：
- 性格不是静态标签，而是可长期演化的数值结构
- 日常交互、工作、旅行、强制唤醒等行为都可能影响这些维度
- 性格值最终会注入 LLM 的 System Prompt，改变宠物表达风格

### 3. HFSM 行为系统
当前规划中的核心状态节点：
- `Idle`
- `Moving`
- `Interacting`
- `Working`
- `Sleeping`

设计目标：
- 宠物行为要能被解释和追踪
- 用户指令与自主行为都进入统一状态机
- 打断、唤醒、工作完成、旅行归来等事件都通过状态切换表达

### 4. 公寓场景
公寓场景是主要的“室内陪伴与家具交互”空间。

设计目标：
- 2D 俯视 / 正交
- Tilemap + CompositeCollider2D 组织房间结构
- 家具可摆放
- 宠物可自主寻路并和家具交互

### 5. 桌面 Overlay
Overlay 场景是“系统陪伴”形态。

设计目标：
- 透明无边框窗口
- 宠物驻留在桌面边缘
- 根据前台应用上下文主动发起建议或对话

### 6. V-Decor 家具系统
家具不是纯装饰，而是玩法系统的一部分。

设计目标：
- 长按 `V` 进入建造模式
- 支持地板家具和贴墙家具
- 家具拥有 Buff
- 家具交互会影响宠物状态与性格
- 摆放结果要能影响寻路

### 7. 聊天 / 工作模式
宠物与 OpenClaw Gateway 的交互至少分成两类：

#### 聊天模式
- 用户发送消息
- 网关返回文本
- 宠物不一定切换状态
- 响应以聊天气泡形式呈现

#### 工作模式
- 用户发出工作型指令
- 宠物切入 `Working`
- 走向工位
- 等待网关返回完成事件

### 8. 旅行系统
旅行是“宠物离场后异步带回内容”的玩法分支。

设计目标：
- 用户可发送“去旅游”一类指令
- 宠物暂时离开场景
- Gateway 异步生成旅行小记与照片
- 宠物归来时带回奖励和展示内容

## 玩家核心循环
1. 查看宠物当前状态与心情
2. 与宠物聊天，或下发工作 / 旅行 / 行为指令
3. 调整公寓中的家具布局与环境 Buff
4. 观察宠物状态、性格与表现变化
5. 在公寓形态和桌面形态之间持续建立长期陪伴关系

## 当前明确写进规则的设计偏好
- 优先“可解释状态机”而不是黑盒行为树
- 优先“数据驱动配置”而不是散落的魔法数字
- 优先“环境反馈型互动”而不是纯按钮触发
- 优先“长期陪伴与记忆感”而不是一次性小玩法

## 当前尚未明确或待后续细化的部分
- 多宠物是否支持
- 养成节奏与数值平衡细则
- 货币、任务、成长解锁等外围系统
- 最终的 UI 视觉风格与完整交互稿
- 网关协议细节和错误恢复 UX
