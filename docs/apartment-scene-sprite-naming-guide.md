## 总规则

### 1. 先分清是不是“家具”
命名第一层先区分两类：

- `家具_...`
  说明这是独立可引用、可能参与交互或配置定义的物件资源
- `环境_...`
  说明这是场景背景、墙面、地板、大块组合层，不作为单独家具逻辑使用

不要把场景背景层命名成 `家具_...`，也不要把真正要交互的床、书桌、竖琴命名成 `环境_...`。

### 2. 家具资源必须带“系统类别词”
对要进入家具系统的 Sprite，命名里必须出现下面这些中文类别词之一：

- `床`
- `工作桌`
- `休闲`
- `竖琴`
- `装饰`

推荐原因：

- 这几个词和当前 `FurnitureCategory` 推断逻辑直接对应
- 后面接 `FurnitureDefinitionSO`、`FurnitureService`、宠物自主行为时最稳

### 3. 不要只写位置，不写物品
不推荐：

- `公寓场景_右中高柜_03`
- `公寓场景_中下小摆件_32`

推荐：

- `家具_装饰_书柜_天使_01`
- `家具_装饰_便签纸_天使_01`

## 推荐格式

### 家具 / 可交互物件
格式：

```text
家具_{类别}_{物体}_{主题}_{编号}
```

示例：

- `家具_床_天使_01`
- `家具_工作桌_天使_01`
- `家具_竖琴_天使_01`
- `家具_装饰_镜子_天使_01`
- `家具_装饰_书柜_天使_01`

说明：

- `类别` 必须是系统能读懂的类别词
- `物体` 是物体本体名
- `主题` 当前公寓场景建议统一先用 `天使`
- `编号` 用两位或三位编号都可以，但同类里要统一

### 纯环境层 / 组合背景层
格式：

```text
环境_{场景}_{区域}_{角色}_{编号}
```

示例：

- `环境_公寓_左侧组合主层_01`
- `环境_公寓_右侧组合主层_01`
- `环境_公寓_地板主层_01`
- `环境_公寓_背景墙主层_01`

说明：

- `场景` 当前统一用 `公寓`
- `区域` 表示左右/上中下等大区域
- `角色` 表示 `主层 / 背景 / 墙面 / 地板 / 组合层`

## 与当前代码类别的对应关系

| 系统类别 | 命名中建议出现的词 | 适合的物体 |
| :--- | :--- | :--- |
| `FurnitureCategory.Bed` | `床` | 床、睡眠家具 |
| `FurnitureCategory.WorkDesk` | `工作桌`、`工作台`、`书桌` | 书桌、工作台 |
| `FurnitureCategory.Leisure` | `休闲`、`竖琴` | 竖琴、娱乐家具 |
| `FurnitureCategory.Decoration` | `装饰`、`床头柜`、`书柜`、`镜子`、`植物`、`摆件` | 镜子、植物、摆件、书柜、壁饰、床头柜 |

注意：

- 如果你把床命名成 `家具_睡觉处_天使_01`，当前代码未必会把它识别成 `床`
- 如果你把书桌命名成 `家具_桌子_天使_01`，当前代码也未必会把它识别成 `工作桌`

所以在现阶段，**类别词要优先满足代码识别，而不是只追求自然语言好看。**

## 命名时的判断顺序

给一个 Sprite 改名时，按这个顺序判断：

1. 它是不是单独的家具或可摆放物件  
是：用 `家具_...`

2. 它是不是纯背景、墙面、地板、大块组合层  
是：用 `环境_...`

3. 如果是家具，它属于哪种系统类别  
优先从 `床 / 工作桌 / 休闲 / 竖琴 / 装饰` 里选

4. 再补物体本体名、主题名、编号
